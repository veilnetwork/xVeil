## Data model

CONCRETE LOG-ENTRY RECORD (the unit stored in Ns.messageLog, JSON, replacing today's `{id,c,d,b,t,s,e?,fi?,fn?}` + the `op:'status'/'del'` shapes):

Forward EVENT row (kind-tagged, carries durable (author,seq)):
```
{
  'id': String,        // message id (uuid.v4 for posts; target-derived not needed)
  'c' : String,        // conversationId = peer node-id hex (server-authenticated)
  'au': String,        // AUTHOR node-id hex (R1/R4 first-class) — NOT inferred from direction
  'sq': int,           // per-conversation per-author seq (R4/R5/R10), gap-free Lamport
  'k' : int,           // EventKind.index  (post|edit|delete|void_|filePost)
  't' : int,           // ts = sentAtMs (display only; fold is ts-free)
  'd' : int,           // MessageDirection.index (kept: cheap local sender/receiver discriminator, derivable from au==me)
  if post/filePost:  'b': String body
  if filePost:       'fd': {name,size,count,blobKind}   // descriptor (reuses fileMetaEnvelope shape)
  if edit:           'tg': targetId, 'b': newBody
  if delete/void_:   'tg': targetId                     // NO body (scrubbed); void_ also drops 'id' on the WIRE only
  if post w/ file:   'fi': fileId, 'fn': fileName        // legacy file fields kept for filePost local read-model
  if edited (folded):'e': 1
}
```
The non-seq SIDE-CHANNEL row stays as-is: `{'op':'status','id','c','s'}` (delivery status is local-only, never an event, never seq-bearing).

KEY: the fold's live state is keyed by (conv, author, seq) — a Dart record `(String conv, String au, int sq)` — NOT by reused log_id. The DURABLE storage key is the §15.4 prefix-packed `log_id = (conv_slot << 40) | conv_seq`.

NEW Dart types:
- `lib/domain/event.dart`: `enum EventKind { post, edit, delete, void_, filePost }` (append-only; react/pin deferred) and a value record `typedef LogEvent = ({EventKind kind, String author, int seq, String id, String? target, String? body, int ts});` plus `Map<String,dynamic> encodeEventBody(LogEvent)` / `LogEvent? decodeEventBody(Map)` for the wire.
- `lib/domain/chat.dart` Message: add `final String? author;` (node hex) and `final int? seq;` + extend `copyWith` to thread them. author is bound at the storage boundary from authenticated m.src (R1), never only from direction.

§15.4 PREFIX LAYOUT in the SINGLE Ns.messageLog (no new namespace — §14.2):
- `static const int _seqBits = 40;` → low 40 bits = conv_seq (1T/conv), high 24 bits = conv_slot (16M convs).
- conv_slot map lives in Ns.settings alongside today's `msg_next_id`/`contacts:index`: KV keys `conv_slot:<convHex>` -> slot int, plus a `conv_slot_next` counter (slot 0 RESERVED as the legacy range so any pre-event-log rows fold without collision).
- per-conversation seq counters in Ns.settings: `conv_seq:<convHex>:<authorHex>` -> next seq for that (conv,author). (Per-author because two authors share a conversation's seq space but each has its OWN gap-free stream — R10/R4.)

## Fold

THE FOLD (rewrite of `_foldCritical` at hidden_volume_storage.dart:724-813) becomes a (conv,author,seq) VALUE-FOLD, two layers per §15.1 R-ORDER. It is ts-FREE for apply; ts is used only at display.

State maps re-keyed from composite `convid` (_msgKey) to a `(conv,au,seq)` apply-key, but the user-visible identity stays the message `id` (display + dedup). Maintain:
- `_evById: Map<(conv,id), Message>` — current folded message (carries author+seq).
- `_evHeld: Map<(conv,targetId), List<HeldOp>>` — edit/delete whose target hasn't arrived; each HeldOp binds its AUTHORIZING author at hold time (R16): `({EventKind k, String author, int seq, String? body})`.
- `_winningEditSeq: Map<(conv,id), int>` — last applied edit seq, for the R5 strictly-greater gate.
- `_scanDeletedKeys` / `_scanDeletedLegacyIds` — kept (born-delete suppression, isMessageDeleted; R17).
- per-author contiguous high-water `_hw: Map<(conv,au), int>` derived during the fold.

ARMS (per decoded row, idempotent on (conv,au,seq)):
- `status` op: unchanged (local delivery status).
- `post`/`filePost`: HARD precondition isMessageDeleted((conv,id))==false (R17) → else drop, still advance high-water. Create/keep `_evById[(conv,id)]` with body/ts/author/seq/direction. Then drain `_evHeld[(conv,id)]`: apply each held op IFF its bound author == this post's author (R16).
- `edit` (target=tg, body=newBody): if target present in `_evById` AND `_evById[(conv,tg)].author == event.author` (R1/R16) AND `event.seq > _winningEditSeq[(conv,tg)] ?? -1` (R5) → replace body, set edited:true, scrub old body chunk via the orphaned-record path, set `_winningEditSeq`. Else if target absent → HOLD (bind author). Else drop.
- `delete`: if target present AND same author → write/keep a body-less VOID slot at the target (R4: the seq slot stays occupied, gap-free), add `(conv,tg)` to `_scanDeletedKeys`, purge file blob (deleteFileOps folded into the same commit). If target absent → HOLD as delete (born-delete on arrival).
- `void_`: clears body but KEEPS the seq slot, marks deleted, ADVANCES high-water; renders nothing (R-VOID — satisfied, replayable, no oracle). An out-of-order void buffers like any gapped event and applies when contiguous.

Fold is idempotent (re-applying same (conv,au,seq) is a no-op given the LWW guards) and convergent.

DERIVED READ METHODS:
- `loadMessages(conv,{limit})` (hidden_volume_storage.dart:348): replace `_scanLog()`+filter with the §15.4 PREFIX RANGE SCAN over [convSlot<<40,(convSlot+1)<<40); fold that window; materialise from `_evById` entries for `conv`; then the DISPLAY sort (replacing the non-stable (timestamp,id) sort at :361-364): STABLE sort by `(effective_ts, author, seq)` where `effective_ts(e) = max(future_clamp(e.ts), running_max_ts of e.author over APPLIED events of seq < e.seq)` — an author-monotone floor that is a pure function of the applied subset (identical on both devices; REJECTS R9 per-device receive-time floor). Keep the existing _wireSentAt future-clamp (messaging.dart:376).
- `loadConversations` (hidden_volume_storage.dart:296): latest-Message-per-conv reduce unchanged in shape; fix its sort (:330) to the same (effective_ts,author,seq) tiebreak if it depends on ordering.
- `isMessageDeleted` (hidden_volume_storage.dart:495): unchanged composite-key membership (R17 hard precondition).

DEDUP: by (author,seq) at apply (idempotent); the message `id` remains the user-facing dedup for the live-arm fast path. TOMBSTONES: id-bearing tombstones (`_scanDeletedKeys`, `op:'del'` local rows) stay LOCAL for born-delete suppression and are FORBIDDEN on the recovery wire — only inert `void_` (no id) goes on the wire (§12.1).

## Sync + gap-fill

WIRE (wire_envelope.dart):

1. APPEND WireKinds (indices 8/9/10/11, never reorder — preserves legacy 0..7):
`enum WireKind { request, accept, message, fileMeta, fileChunk, ack, edit, del, event, sync, reconnect, filePost }`

2. STRUCTURAL MARKER (RULE WC, BLOCKER): event/sync/reconnect/filePost frames carry a reserved `'v':2` key in encode() (lines 48-53). decode() (lines 57-76) must DETECT a `v:2` frame whose `t` an un-upgraded build doesn't know and return a DROP sentinel — never fall through to the plain WireKind.message fallback at line 75. Concretely: if `decoded['v']==2` and `t` is out of THIS build's range → return null (caller drops); also if `v:2` and `t` IS known, decode normally. Regression test: an un-upgraded decoder fed an event/sync frame yields NO rendered message (drop), never a JSON blob shown as chat text.

3. ENVELOPE for event: `WireEnvelope.event(LogEvent)` body = `encodeEventBody` JSON `{k,a:author,q:seq,id,tg?,b?,ts}`, `v:2`. On decode in _dispatch: BIND `author := m.src.hex`; REJECT if in-band `a != m.src.hex` (R1).

4. SYNC frame: `WireEnvelope.sync(...)` body = `{hw:{<authorHex>:contiguousHighWater}, holes:{<authorHex>:[[lo,hi],...]}, ep:epoch}`, `v:2`. It is BOTH ack ("I have your events <= hw[me]") AND gap query ("ship me > hw / fill holes"). RULE HW: on receiving a peer's sync, CLAMP its claimed hw about MY author-stream to my_highest_emitted_seq (a peer cannot ack a seq it was never owed); derive MessageStatus (delivered) ONLY from the clamped, contiguous, monotone high-water (replaces the per-id ack arm at messaging.dart:592-623). RULE NH: advertise contiguous hw AND explicit holes; a named-hole repair is EXEMPT from the coarse per-peer reconcile rate-limiter but governed per-hole by `ep` (sync-epoch/nonce) + an `answeredHole[(peer,author,range)]` ledger → answer a given hole ONCE per epoch.

5. GAP-FILL: reuse the sync frame as the request (low hw / named holes ARE the request). On receiving sync, ship every local event with seq > peer hw[author], oldest-first, BOUNDED <=N/round (R7), via WireEnvelope.event over live _send AND _maybeStash. A replayed void_ SATISFIES a hole (advances hw, never re-requested). An out-of-order event with seq>hw+1 buffers in a bounded gap buffer (replaces _pendingOps) and queues a hole.

6. R-DEP DEPOSIT: each contiguous round OR named-hole repair is its OWN opaque mailbox blob under a FRESH content-id via `_contentIdFor(tag)` (messaging.dart:1112), tag = a round nonce or `'hole:<author>:<lo>-<hi>:<epoch>'`. This is exactly the shipped 'edit:'/'del:' put-new/ack-old pattern (content_id already deterministic-domain-separated → R12 already shipped). Cap distinct in-flight ids per (recipient,author,epoch) + jitter, reusing `_stashed`/`_stashRetryBackoff`/`_peerUnresolvedBackoff`. Receiver dedups/applies the INNER (author,seq) on drain; a blob is acked or TTL-dropped once its seq range is covered.

7. IDEMPOTENT FOLD FROM LIVE OR MAILBOX: `deliverInbound`==`_onInbound` already routes mailbox drain through the SAME serialized `_inboundChain` (messaging.dart:458-480) as live → the event arm folds identically from either source; idempotency on (author,seq) makes double-delivery safe.

8. RECONNECT (R-RECOVER2 + §15.7): authenticated WireKind.reconnect fired when a message stays un-acked past ~2 min, BOUNDED ~5-6 tries/~1-2h then "not delivered"+manual retry. An accepted contact's hw DROPPING below a previously-acked floor = WIPE signal → reconcile-from-zero. Receiver disambiguates by OWN state (blocked→silent drop; unknown/pending→pendingIncoming under kMaxPreConsentIntros; accepted→reconcile). Case B (identity lost→new node_id) = manual re-invite, shown honestly.

9. CONSENT (R2): event/sync from a NON-accepted node route through `_capPreConsentIntros` (messaging.dart:407) + `_inboundChain` → materialize AT MOST ONE pending intro (latest post body) under kMaxPreConsentIntros=5; NEVER apply edit/delete/status, NEVER allocate gap/pending buffers, NEVER drive multi-event replay. Full reconcile fires ONLY after ContactStatus.accepted (messaging.dart:400,485,495). Blocked → drop silently.

10. CAPABILITY PIN (RULE WC + §12.8/R3): emit event/sync ONLY to a peer pinned event-capable per (contact, acceptance-epoch); reset only on authenticated re-accept carrying the capability advert. Never mix legacy edit/del/message with event frames in one event-capable accepted conversation (prevents a downgrade replaying a legacy edit that bypasses R1). R17: a mixed pair degrades to today's best-effort path until BOTH ends run the protocol.

## Deniability / compaction / media

COMPACTION / FORWARD-SECRECY (R6 + R-SCRUB2, §12.3/§14.3/§15.1):

NORMATIVE RULE: any EVER-EDITABLE event is committed ONE-RECORD-PER-DataBatch (one per-op commit) on EVERY path — live, edit, delete, AND recovery replay. This is the EXISTING per-op default: appendMessage (hidden_volume_storage.dart:386-392), editMessage (:445-447), deleteMessage (:477-480) are each their own single-op commit → its own singleton DataBatch. The new appendEvent/edit/void paths MUST preserve this and MUST NOT coalesce multiple editable rows into one commit.

The FS hole exists ONLY in BATCHED paths: the N-events/round recovery replay, and the existing coalesced `_tombstoneAllOps`/`removeConversation` (hidden_volume_storage.dart:512-533). Rule: apply bulk recovery as ONE DENSE multi-record commit ONLY for historical NO-LONGER-EDITABLE posts; on the first edit/delete touching a dense historical batch, REPACK just that batch (or write the edit as its own fresh one-record commit + scrub the orphan). STRIKE R6's container-wide compact_known fallback (it keeps only the supplied-password space and DROPS every decoy/duress space = a duress break) — reserve it for the single-identity all-handles-closed Storage→Compact UI only. NOTE Stage-3 compact_known re-coalesces pages into shared batches → REOPENS the hole after a user compacts → the messenger must keep editable records one-per-batch.

R6 SHIP-GATE REGRESSION TEST (does not exist today, would FAIL today): edit/delete ONE record packed in a MULTI-record DataBatch → assert the orphan plaintext chunk is gone after `scrubDeleted` → `vacuum_data_batches` (scrubDeleted at hidden_volume_storage.dart:577-586; vacuum erases a DataBatch only when NONE of its packed log-ids remain referenced).

PER-IDENTITY STORAGE POLICY (§16.3-PI, load-bearing): media tier policy lives in each identity's OWN space settings (Ns.settings), NEVER global; sync high-waters are sealed inside opaque mailbox blobs (relay learns nothing); a decoy/duress identity's log + clocks live in its own space (per-space isolation).

DENIABILITY BOUNDS (R10/R11): per-conversation seq (R10) bounds the seized-device event-count leak to ONE chat and prevents gap-fill seeing other-chat holes. R11: a gap-free per-author seq leaks the COUNT of scrubbed/superseded mutations — DECISION: accept + bound via R10 + DOCUMENT that the seq ceiling reveals lifetime-mutation-count, not just current message count.

MEDIA TIERING (§16) — WHAT TO DEFER: v1 = IN-VOLUME blobs ONLY (Ns.fileChunks, scrubbable, hidden by lock+decoy/duress) — the DEFAULT and only shipped tier. ON-DISK ENCRYPTED (tier 2: blob AEAD under a per-file key on plain FS, container holds only descriptor + key; confidentiality kept, deniability degraded; forward-secrecy-on-delete = scrub the in-volume per-file key) and ON-DISK PLAINTEXT (tier 3) are DEFERRED behind a focused deniability review (§16.5) BEFORE any code. Media KINDS (image=full blob + always-in-volume <=64KB thumb; video=+poster+durationMs; sticker=pack:id reference to a deduped cached asset; emoji=inline Unicode text, emoji-reactions a post-v1 event kind) are also DEFERRED. v1 ships only the generic `filePost` (descriptor + chunks). IndexFull relief + the eraseSpace→list_namespaces deniability fix are DEFERRED until a real user nears the cap (NO veil work for v1).
